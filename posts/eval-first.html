Eval-first: why every agent I ship gets a scorecard

“It feels better” is not a merge criterion. The first thing I build for an agent isn’t the agent — it’s the way I’ll know whether a change made it better or worse.

Start with a tiny set

Twenty realistic tasks with known-good outcomes beat a thousand synthetic ones. Pull them from real usage the moment you have any, and keep the hard cases you almost got wrong.

Score what matters

  • Task success — did it actually do the thing?
  • Tool accuracy — right tool, right arguments, no thrash.
  • Cost — tokens and latency per task, because that ships too.
No number, no merge. The scorecard is the contract between “I changed a prompt” and “it’s allowed to reach users.”

Run it on every change

Wire the eval into CI so a regression fails the build the same way a broken test would. An LLM-as-judge is fine for the fuzzy parts, as long as you’ve checked the judge against your own labels first.

Eval-first is slower on day one and much faster by week three, because you stop guessing.
