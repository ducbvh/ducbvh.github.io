Making agents reliable: the loop beats the prompt

Most agent demos are one clever prompt away from falling over. They work in the recording and break on the second real input. The fix is rarely a better prompt — it’s a better loop.

An agent that works once is a prototype. An agent that works when the input gets weird is a product.

Plan, act, verify, retry

The shape that has held up for me is boring on purpose: the agent writes a short plan, takes one step, checks the result against the goal, and either continues or retries. Structure carries the reliability that a single prompt can’t.

plan  → act → verify ─┐
   ↑                  │
   └── retry ◀── fail ─┘

Make failure cheap

  • Bound every step: max tool calls, max retries, a wall-clock budget.
  • Prefer small reversible actions over one big irreversible one.
  • Log the full trace — plan, tool calls, and the verifier’s verdict.

The verifier earns its keep

A cheap check after each step catches the drift that a final answer hides. It doesn’t need to be another model — often a schema check, a test, or a grep is enough. The point is that the agent notices it’s wrong before you do.

None of this is glamorous. That’s the point: production is the job, and reliability is built, not prompted.
